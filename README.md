# 🚀 Микросервис Test Service

Простой и масштабируемый микросервис для управления тестовыми данными, готовый к работе с Docker Compose, Docker Desktop Kubernetes и облачным развертыванием.

## 🎯 Особенности

- ✅ **REST API** для управления тестовыми сущностями
- ✅ **PostgreSQL 15** с автоматическими миграциями
- ✅ **Docker Compose** для локальной разработки
- ✅ **Docker Desktop Kubernetes** для локального K8s
- ✅ **Kubernetes-ready** архитектура для продакшена
- ✅ **Health checks** для мониторинга
- ✅ **Готовая миграция** в Yandex Cloud
- ✅ **Автоматические бэкапы** базы данных
- ✅ **Полная CI/CD** система

## 🔐 Настройка Google OAuth

### Получение Google Client ID

Для работы авторизации через Google необходимо настроить OAuth credentials:

1. Перейдите в [Google Cloud Console](https://console.cloud.google.com/)
2. Создайте новый проект или выберите существующий
3. Перейдите в раздел **"APIs & Services"** → **"Credentials"**
4. Нажмите **"Create Credentials"** → **"OAuth client ID"**
5. Выберите тип приложения **"Web application"**
6. Добавьте authorized redirect URIs:
   - `http://localhost:3001` (для локальной разработки)
   - Ваш production URL (например, `https://yourdomain.com`)
7. Скопируйте **Client ID**

### Настройка переменных окружения

1. Откройте файл `.env` в корне проекта
2. Замените placeholder значение на реальный Client ID:
   ```env
   GOOGLE_CLIENT_ID=ваш_реальный_google_client_id.apps.googleusercontent.com
   GOOGLE_CLIENT_SECRET=ваш_google_client_secret
   ```
3. Также обновите `.env.production` для production окружения

⚠️ **Важно**: Без настройки реального Google Client ID авторизация через Google работать не будет. При попытке входа вы увидите ошибку **"The OAuth client was not found"**.

### Проверка настройки

После настройки переменных окружения:

1. Перезапустите приложение:
   ```bash
   docker-compose down
   docker-compose up -d
   ```

2. Откройте user-portal: `http://localhost:3001`

3. На вкладке **"Вход"** должна работать кнопка **"Войти через Google"**

### Безопасность

- **Client ID** - публичное значение, можно хранить в `.env`
- **Client Secret** - секретное значение, используется только на сервере
- Никогда не коммитьте реальные значения в git
- Используйте разные credentials для development и production

## 📧 Настройка способов подтверждения регистрации

Приложение поддерживает два способа подтверждения регистрации пользователей:

### 1. Подтверждение по Email (по умолчанию)

При регистрации пользователь может выбрать подтверждение через email:

- Пользователь вводит email адрес в форме регистрации
- Система отправляет 6-значный код подтверждения на указанный email
- Пользователь вводит полученный код в форме
- При успешной проверке кода завершается регистрация

**Настройки в `.env`:**
```env
# Email подтверждение
EMAIL_VERIFICATION_ENABLED=true
EMAIL_CODE_LENGTH=6
EMAIL_CODE_EXPIRATION=10
EMAIL_MAX_SENDS_PER_HOUR=3
```

**Описание настроек:**
- `EMAIL_VERIFICATION_ENABLED` - включить/выключить подтверждение по email
- `EMAIL_CODE_LENGTH` - длина кода подтверждения (рекомендуется 6)
- `EMAIL_CODE_EXPIRATION` - срок действия кода в минутах (по умолчанию 10)
- `EMAIL_MAX_SENDS_PER_HOUR` - максимальное количество отправок кодов в час на один email

### 2. Подтверждение по SMS

Альтернативный способ подтверждения через SMS:

- Пользователь выбирает "Подтверждение по телефону" в форме регистрации
- Вводит номер телефона в международном формате (+7XXXXXXXXXX)
- Система отправляет SMS с кодом подтверждения
- Пользователь вводит полученный код в форме
- При успешной проверке кода завершается регистрация

**Настройки SMS провайдера в `.env`:**
```env
# SMS подтверждение
SMS_PROVIDER_API_KEY=ваш_api_ключ_sms_провайдера
SMS_PROVIDER_API_URL=https://api.sms-provider.com
PHONE_CODE_LENGTH=6
PHONE_CODE_EXPIRATION=10
PHONE_MAX_SENDS_PER_HOUR=3
```

**Описание настроек:**
- `SMS_PROVIDER_API_KEY` - API ключ вашего SMS провайдера
- `SMS_PROVIDER_API_URL` - URL API вашего SMS провайдера
- `PHONE_CODE_LENGTH` - длина SMS кода (рекомендуется 6)
- `PHONE_CODE_EXPIRATION` - срок действия SMS кода в минутах
- `PHONE_MAX_SENDS_PER_HOUR` - лимит отправок SMS в час на один номер

### Выбор способа подтверждения

В форме регистрации пользователь видит переключатель (RadioGroup) с двумя опциями:
- **"Подтверждение по email"** - выбрано по умолчанию
- **"Подтверждение по телефону"** - альтернативный способ

При выборе способа подтверждения:
- Отображается только соответствующее поле (email ИЛИ телефон)
- Валидация формы проверяет только активное поле
- Логика отправки кода адаптируется под выбранный способ

### Безопасность подтверждения

Для защиты от злоупотреблений реализованы следующие меры:

- **Rate limiting**: Ограничение количества отправок кодов (3 в час по умолчанию)
- **Срок действия**: Коды действительны только 10 минут
- **Валидация**: Проверка формата email и телефона на клиенте и сервере
- **Повторная отправка**: Возможность запросить новый код при истечении срока

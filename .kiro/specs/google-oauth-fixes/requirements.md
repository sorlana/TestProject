# Документ требований: Исправление Google OAuth

## Введение

Данный документ описывает требования к исправлению проблем с Google OAuth авторизацией в приложении user-portal. Необходимо устранить две основные проблемы: удалить избыточную кнопку Google OAuth на вкладке регистрации и обновить документацию по настройке реального Google Client ID.

## Глоссарий

- **User_Portal**: React приложение для пользовательского интерфейса, расположенное в `apps/user-portal/`
- **LoginPage**: Страница авторизации и регистрации с табами "Вход" и "Регистрация"
- **GoogleAuthButton**: Компонент кнопки для авторизации через Google OAuth
- **Google_Client_ID**: Идентификатор клиента Google OAuth, необходимый для работы авторизации через Google
- **Registration_Tab**: Вкладка "Регистрация" на странице LoginPage
- **Login_Tab**: Вкладка "Вход" на странице LoginPage
- **Environment_File**: Файл `.env` с переменными окружения проекта
- **RegisterForm**: Компонент формы регистрации пользователя
- **Verification_Method**: Способ подтверждения регистрации (email или телефон)
- **Email_Verification**: Процесс подтверждения регистрации через email с кодом
- **Phone_Verification**: Процесс подтверждения регистрации через SMS с кодом
- **Verification_Code**: Код подтверждения, отправляемый пользователю для верификации
- **RadioGroup**: Компонент выбора одного варианта из нескольких (email или телефон)

## Требования

### Требование 1: Удаление Google OAuth с вкладки регистрации

**User Story:** Как разработчик, я хочу удалить кнопку Google OAuth с вкладки регистрации, чтобы упростить интерфейс и избежать дублирования функциональности.

#### Критерии приемки

1. WHEN пользователь открывает вкладку "Регистрация", THE User_Portal SHALL отображать только форму регистрации без кнопки Google OAuth
2. WHEN пользователь открывает вкладку "Регистрация", THE User_Portal SHALL NOT отображать разделитель (Divider) перед несуществующей кнопкой Google OAuth
3. THE LoginPage SHALL удалить блок GoogleAuthButton из панели Registration_Tab (строки 133-139)
4. THE LoginPage SHALL удалить Divider перед GoogleAuthButton из панели Registration_Tab (строка 130)

### Требование 2: Сохранение Google OAuth на вкладке входа

**User Story:** Как пользователь, я хочу иметь возможность войти через Google на вкладке "Вход", чтобы использовать быструю авторизацию.

#### Критерии приемки

1. WHEN пользователь открывает вкладку "Вход", THE User_Portal SHALL отображать кнопку "Войти через Google"
2. THE LoginPage SHALL сохранить GoogleAuthButton на панели Login_Tab без изменений
3. THE LoginPage SHALL сохранить Divider перед GoogleAuthButton на панели Login_Tab

### Требование 3: Обновление документации по настройке Google Client ID

**User Story:** Как разработчик, я хочу иметь четкую документацию по настройке Google Client ID, чтобы правильно настроить OAuth авторизацию.

#### Критерии приемки

1. THE Environment_File SHALL содержать placeholder значение `GOOGLE_CLIENT_ID=your_google_client_id` для примера
2. THE Documentation SHALL содержать инструкции по получению реального Google Client ID
3. THE Documentation SHALL содержать инструкции по замене placeholder значения на реальный Client ID
4. THE Documentation SHALL содержать ссылку на Google Cloud Console для создания OAuth credentials
5. THE Documentation SHALL содержать предупреждение о том, что без реального Client ID авторизация через Google работать не будет

### Требование 4: Структура кода после изменений

**User Story:** Как разработчик, я хочу иметь чистый и понятный код, чтобы легко поддерживать приложение.

#### Критерии приемки

1. THE LoginPage SHALL содержать только один экземпляр GoogleAuthButton (на вкладке "Вход")
2. THE LoginPage SHALL сохранить все комментарии и структуру кода для вкладки "Вход"
3. THE LoginPage SHALL сохранить RegisterForm на вкладке "Регистрация" без изменений
4. THE LoginPage SHALL сохранить все обработчики событий и хуки без изменений

### Требование 5: Выбор способа подтверждения регистрации

**User Story:** Как пользователь, я хочу выбрать способ подтверждения регистрации (email или телефон), чтобы использовать удобный для меня метод верификации.

#### Критерии приемки

1. WHEN пользователь открывает форму регистрации, THE RegisterForm SHALL отображать RadioGroup с двумя опциями: "Подтверждение по email" и "Подтверждение по телефону"
2. WHEN пользователь выбирает "Подтверждение по email", THE RegisterForm SHALL скрывать поле ввода телефона
3. WHEN пользователь выбирает "Подтверждение по email", THE RegisterForm SHALL отображать поле ввода email
4. WHEN пользователь выбирает "Подтверждение по телефону", THE RegisterForm SHALL отображать поле ввода телефона
5. WHEN пользователь выбирает "Подтверждение по телефону", THE RegisterForm SHALL скрывать поле ввода email
6. THE RegisterForm SHALL по умолчанию выбирать "Подтверждение по email"

### Требование 6: Подтверждение регистрации через email

**User Story:** Как пользователь, я хочу подтвердить регистрацию через email, чтобы верифицировать свою учетную запись.

#### Критерии приемки

1. WHEN пользователь заполняет форму регистрации с выбранным "Подтверждение по email" и нажимает "Зарегистрироваться", THE System SHALL отправить код подтверждения на указанный email
2. WHEN код подтверждения отправлен, THE RegisterForm SHALL отображать поле "Код подтверждения" для ввода кода
3. WHEN пользователь вводит код подтверждения, THE System SHALL проверить соответствие введенного кода отправленному коду
4. WHEN введенный код совпадает с отправленным кодом, THE System SHALL завершить регистрацию и отправить письмо об успешной регистрации
5. WHEN введенный код не совпадает с отправленным кодом, THE System SHALL отобразить сообщение об ошибке "Неверный код подтверждения"
6. THE System SHALL установить срок действия кода подтверждения 10 минут
7. WHEN срок действия кода истек, THE System SHALL отобразить сообщение об ошибке и предложить отправить код повторно

### Требование 7: Подтверждение регистрации через телефон

**User Story:** Как пользователь, я хочу подтвердить регистрацию через SMS, чтобы использовать телефон для верификации.

#### Критерии приемки

1. WHEN пользователь заполняет форму регистрации с выбранным "Подтверждение по телефону" и нажимает "Зарегистрироваться", THE System SHALL использовать существующую логику подтверждения по телефону
2. THE System SHALL отправить SMS с кодом подтверждения на указанный номер телефона
3. THE System SHALL использовать существующие настройки из Environment_File для SMS провайдера
4. THE System SHALL применять существующие ограничения на количество отправок SMS (PHONE_MAX_SENDS_PER_HOUR)
5. THE System SHALL применять существующий срок действия кода (PHONE_CODE_EXPIRATION)

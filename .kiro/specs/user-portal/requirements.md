# Документ требований: User Portal

## Введение

User Portal - это фронтенд-микросервис на базе React и Blueprint UI, предоставляющий пользовательский интерфейс для авторизации, регистрации и доступа к личному кабинету. Микросервис интегрируется с существующим Auth Service через HTTP API и использует JWT токены для аутентификации.

## Глоссарий

- **User_Portal**: Фронтенд SPA приложение на React с Blueprint UI
- **Auth_Service**: Существующий микросервис аутентификации на ASP.NET Core 8.0
- **JWT_Token**: JSON Web Token для аутентификации пользователя
- **Protected_Route**: Маршрут, доступный только авторизованным пользователям
- **Login_Form**: Форма входа в систему
- **Registration_Form**: Форма регистрации нового пользователя
- **Dashboard**: Личный кабинет пользователя
- **Token_Interceptor**: Механизм автоматического добавления JWT токена в HTTP запросы
- **Blueprint_UI**: Библиотека UI компонентов от Palantir

## Требования

### Требование 1: Страница авторизации

**User Story:** Как пользователь, я хочу войти в систему или зарегистрироваться, чтобы получить доступ к личному кабинету.

#### Критерии приемки

1. WHEN пользователь открывает /app/login, THE User_Portal SHALL отобразить страницу с двумя табами: "Вход" и "Регистрация"
2. WHEN пользователь находится на табе "Вход", THE Login_Form SHALL содержать поля: логин/email, пароль и чекбокс "Запомнить меня"
3. WHEN пользователь находится на табе "Регистрация", THE Registration_Form SHALL содержать поля: логин, email, телефон, пароль, подтверждение пароля и ФИО
4. WHEN пользователь нажимает кнопку "Войти через Google", THE User_Portal SHALL инициировать OAuth авторизацию через Google
5. WHEN пользователь успешно авторизуется или регистрируется, THE User_Portal SHALL выполнить редирект на /app/dashboard

### Требование 2: Валидация форм на клиенте

**User Story:** Как пользователь, я хочу получать мгновенную обратную связь о корректности введенных данных, чтобы исправить ошибки до отправки формы.

#### Критерии приемки

1. WHEN пользователь вводит email, THE User_Portal SHALL проверить соответствие формату email
2. WHEN пользователь вводит пароль в форме регистрации, THE User_Portal SHALL проверить минимальную длину 8 символов
3. WHEN пользователь вводит подтверждение пароля, THE User_Portal SHALL проверить совпадение с основным паролем
4. WHEN валидация не пройдена, THE User_Portal SHALL отобразить сообщение об ошибке через Blueprint Callout компонент
5. WHEN все поля валидны, THE User_Portal SHALL активировать кнопку отправки формы

### Требование 3: Интеграция с Auth Service

**User Story:** Как система, я хочу взаимодействовать с Auth Service для аутентификации пользователей, чтобы обеспечить безопасный доступ.

#### Критерии приемки

1. WHEN пользователь отправляет форму входа, THE User_Portal SHALL отправить POST запрос на /api/auth/login с учетными данными
2. WHEN пользователь отправляет форму регистрации, THE User_Portal SHALL отправить POST запрос на /api/auth/register с данными пользователя
3. WHEN пользователь нажимает "Войти через Google", THE User_Portal SHALL отправить POST запрос на /api/auth/google с OAuth токеном
4. WHEN Auth_Service возвращает JWT токены, THE User_Portal SHALL сохранить их в localStorage или sessionStorage
5. WHEN JWT токен истекает, THE User_Portal SHALL автоматически отправить POST запрос на /api/auth/refresh для обновления токена
6. WHEN пользователь выходит из системы, THE User_Portal SHALL отправить POST запрос на /api/auth/logout

### Требование 4: Управление JWT токенами

**User Story:** Как система, я хочу безопасно хранить и использовать JWT токены, чтобы поддерживать сессию пользователя.

#### Критерии приемки

1. WHEN пользователь отмечает чекбокс "Запомнить меня", THE User_Portal SHALL сохранить JWT токены в localStorage
2. WHEN пользователь не отмечает чекбокс "Запомнить меня", THE User_Portal SHALL сохранить JWT токены в sessionStorage
3. WHEN User_Portal выполняет HTTP запрос к защищенному API, THE Token_Interceptor SHALL автоматически добавить JWT токен в заголовок Authorization
4. WHEN JWT токен истекает, THE User_Portal SHALL автоматически обновить токен через /api/auth/refresh
5. WHEN обновление токена не удается, THE User_Portal SHALL удалить токены и выполнить редирект на /app/login

### Требование 5: Личный кабинет

**User Story:** Как авторизованный пользователь, я хочу видеть личный кабинет с навигацией, чтобы управлять своим профилем и данными.

#### Критерии приемки

1. WHEN авторизованный пользователь открывает /app/dashboard, THE User_Portal SHALL отобразить страницу Dashboard с header и основной областью
2. WHEN Dashboard отображается, THE User_Portal SHALL показать в header логотип/название платформы слева
3. WHEN Dashboard отображается, THE User_Portal SHALL показать в header логин пользователя как ссылку справа
4. WHEN Dashboard отображается, THE User_Portal SHALL показать в header кнопку "Выйти" справа
5. WHEN пользователь нажимает кнопку "Выйти", THE User_Portal SHALL вызвать API /api/auth/logout, удалить токены и выполнить редирект на /app/login

### Требование 6: Защита маршрутов

**User Story:** Как система, я хочу ограничить доступ к защищенным страницам только для авторизованных пользователей, чтобы обеспечить безопасность.

#### Критерии приемки

1. WHEN неавторизованный пользователь пытается открыть /app/dashboard, THE Protected_Route SHALL выполнить редирект на /app/login
2. WHEN неавторизованный пользователь пытается открыть любой защищенный маршрут, THE Protected_Route SHALL проверить наличие валидного JWT токена
3. WHEN JWT токен отсутствует или истек, THE Protected_Route SHALL выполнить редирект на /app/login
4. WHEN JWT токен валиден, THE Protected_Route SHALL разрешить доступ к запрашиваемой странице
5. WHEN пользователь открывает /app без авторизации, THE User_Portal SHALL выполнить редирект на /app/login
6. WHEN авторизованный пользователь открывает /app, THE User_Portal SHALL выполнить редирект на /app/dashboard

### Требование 7: Маршрутизация

**User Story:** Как пользователь, я хочу перемещаться между страницами приложения, чтобы использовать различные функции.

#### Критерии приемки

1. THE User_Portal SHALL поддерживать маршрут /app/login для страницы авторизации/регистрации
2. THE User_Portal SHALL поддерживать маршрут /app/dashboard для личного кабинета
3. THE User_Portal SHALL поддерживать маршрут /app/payment для страницы оплаты (заглушка)
4. WHEN пользователь открывает /app, THE User_Portal SHALL выполнить редирект на /app/dashboard если авторизован, иначе на /app/login
5. THE User_Portal SHALL использовать React Router для управления маршрутизацией

### Требование 8: Обработка ошибок

**User Story:** Как пользователь, я хочу видеть понятные сообщения об ошибках, чтобы понимать, что пошло не так и как это исправить.

#### Критерии приемки

1. WHEN Auth_Service возвращает ошибку, THE User_Portal SHALL отобразить сообщение об ошибке через Blueprint Toast или Callout
2. WHEN происходит ошибка сети (Auth_Service недоступен), THE User_Portal SHALL отобразить сообщение "Сервис временно недоступен"
3. WHEN Auth_Service возвращает ошибку валидации (HTTP 400), THE User_Portal SHALL отобразить конкретные ошибки валидации для соответствующих полей
4. WHEN Auth_Service возвращает ошибку авторизации (HTTP 401), THE User_Portal SHALL удалить токены и выполнить редирект на /app/login
5. WHEN происходит неожиданная ошибка, THE User_Portal SHALL отобразить общее сообщение об ошибке и записать детали в консоль

### Требование 9: Адаптивный дизайн

**User Story:** Как пользователь, я хочу использовать приложение на различных устройствах, чтобы иметь доступ с любого девайса.

#### Критерии приемки

1. WHEN пользователь открывает User_Portal на мобильном устройстве (ширина < 768px), THE User_Portal SHALL отобразить мобильную версию интерфейса
2. WHEN пользователь открывает User_Portal на планшете (ширина 768px - 1024px), THE User_Portal SHALL отобразить планшетную версию интерфейса
3. WHEN пользователь открывает User_Portal на десктопе (ширина > 1024px), THE User_Portal SHALL отобразить десктопную версию интерфейса
4. WHEN размер экрана изменяется, THE User_Portal SHALL адаптировать интерфейс без перезагрузки страницы
5. THE User_Portal SHALL использовать Blueprint UI компоненты, которые поддерживают адаптивность

### Требование 10: Развертывание и инфраструктура

**User Story:** Как DevOps инженер, я хочу развернуть User Portal в контейнере, чтобы обеспечить консистентность окружения и масштабируемость.

#### Критерии приемки

1. THE User_Portal SHALL быть упакован в Docker контейнер с Nginx для раздачи статики
2. WHEN Nginx получает запрос к /api/*, THE User_Portal SHALL проксировать запрос на Auth_Service
3. THE User_Portal SHALL включать Kubernetes манифесты (deployment, service, ingress) в директории k8s/
4. WHEN User_Portal развернут в Kubernetes, THE User_Portal SHALL быть доступен на пути /app/*
5. THE User_Portal SHALL использовать Nginx конфигурацию для обработки SPA маршрутизации (fallback на index.html)

### Требование 11: Безопасность

**User Story:** Как система, я хочу защитить пользовательские данные и предотвратить атаки, чтобы обеспечить безопасность приложения.

#### Критерии приемки

1. THE User_Portal SHALL использовать встроенную защиту React от XSS атак
2. WHEN User_Portal отправляет запросы на изменение данных, THE User_Portal SHALL включать CSRF токен
3. THE User_Portal SHALL хранить JWT токены только в localStorage или sessionStorage (не в cookies без HttpOnly)
4. WHEN User_Portal развернут в продакшене, THE User_Portal SHALL быть доступен только через HTTPS
5. THE User_Portal SHALL не логировать чувствительные данные (пароли, токены) в консоль в продакшене

### Требование 12: Структура проекта

**User Story:** Как разработчик, я хочу иметь четкую структуру проекта, чтобы легко ориентироваться в коде и поддерживать его.

#### Критерии приемки

1. THE User_Portal SHALL располагаться в директории apps/user-portal/
2. THE User_Portal SHALL содержать Dockerfile в корне проекта
3. THE User_Portal SHALL содержать Kubernetes манифесты в директории k8s/
4. THE User_Portal SHALL содержать Nginx конфигурацию для раздачи статики и проксирования API
5. THE User_Portal SHALL использовать Vite для сборки проекта

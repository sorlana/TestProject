# Документ требований: Микросервис аутентификации пользователей

## Введение

Микросервис аутентификации пользователей предоставляет функциональность регистрации, входа и управления пользователями для системы онлайн-обучения. Система поддерживает традиционную аутентификацию по логину и паролю, вход через Google OAuth, подтверждение телефона через SMS и интеграцию с системой подписок nopCommerce.

## Глоссарий

- **Система**: Микросервис аутентификации пользователей
- **Пользователь**: Физическое лицо, использующее систему для регистрации и входа
- **JWT_Токен**: JSON Web Token для аутентификации запросов
- **Refresh_Токен**: Токен для обновления JWT токена без повторного ввода учетных данных
- **SMS_Провайдер**: Внешний сервис для отправки SMS сообщений
- **Google_OAuth**: Протокол OAuth 2.0 для аутентификации через Google
- **nopCommerce**: Платформа электронной коммерции для управления подписками
- **Подписка**: Платный доступ пользователя к функциям системы на определенный период
- **Личный_Кабинет**: Защищенная область системы, доступная только авторизованным пользователям
- **Код_Подтверждения**: Одноразовый код, отправляемый по SMS для подтверждения телефона
- **Identity_Провайдер**: ASP.NET Core Identity для управления пользователями и ролями

## Требования

### Требование 1: Регистрация пользователя по логину и паролю

**Пользовательская история:** Как новый пользователь, я хочу зарегистрироваться в системе по логину и паролю, чтобы получить доступ к функциям платформы.

#### Критерии приемки

1. КОГДА пользователь отправляет форму регистрации с валидными данными (логин, пароль, подтверждение пароля, email, телефон), ТО Система ДОЛЖНА создать новую учетную запись пользователя
2. КОГДА пользователь вводит email, ТО Система ДОЛЖНА проверить, что email соответствует стандартному формату email адреса
3. КОГДА пользователь вводит телефон, ТО Система ДОЛЖНА проверить, что телефон соответствует формату телефонного номера
4. КОГДА пользователь вводит пароль, ТО Система ДОЛЖНА проверить, что пароль содержит минимум 8 символов, как минимум одну заглавную букву и как минимум одну цифру
5. КОГДА пользователь вводит подтверждение пароля, ТО Система ДОЛЖНА проверить, что подтверждение пароля совпадает с паролем
6. КОГДА пользователь пытается зарегистрироваться с уже существующим логином, ТО Система ДОЛЖНА отклонить регистрацию и вернуть сообщение об ошибке
7. КОГДА пользователь пытается зарегистрироваться с уже существующим email, ТО Система ДОЛЖНА отклонить регистрацию и вернуть сообщение об ошибке
8. ГДЕ пользователь предоставляет необязательные поля (имя, фамилия, отчество), ТО Система ДОЛЖНА сохранить эти данные в профиле пользователя

### Требование 2: Подтверждение телефона через SMS

**Пользовательская история:** Как зарегистрированный пользователь, я хочу подтвердить свой номер телефона через SMS код, чтобы завершить регистрацию и обеспечить безопасность аккаунта.

#### Критерии приемки

1. КОГДА пользователь завершает регистрацию, ТО Система ДОЛЖНА сгенерировать шестизначный числовой Код_Подтверждения
2. КОГДА Код_Подтверждения сгенерирован, ТО Система ДОЛЖНА отправить код через SMS_Провайдер на указанный телефон пользователя
3. КОГДА пользователь вводит корректный Код_Подтверждения в течение 10 минут, ТО Система ДОЛЖНА пометить телефон как подтвержденный
4. КОГДА пользователь вводит некорректный Код_Подтверждения, ТО Система ДОЛЖНА отклонить подтверждение и вернуть сообщение об ошибке
5. КОГДА Код_Подтверждения не использован в течение 10 минут, ТО Система ДОЛЖНА пометить код как истекший
6. КОГДА пользователь запрашивает повторную отправку кода, ТО Система ДОЛЖНА сгенерировать новый Код_Подтверждения и отправить его через SMS_Провайдер
7. КОГДА пользователь запрашивает повторную отправку кода более 3 раз в течение часа, ТО Система ДОЛЖНА отклонить запрос и вернуть сообщение об ограничении

### Требование 3: Аутентификация по логину и паролю

**Пользовательская история:** Как зарегистрированный пользователь, я хочу войти в систему по логину и паролю, чтобы получить доступ к своему Личному_Кабинету.

#### Критерии приемки

1. КОГДА пользователь отправляет корректные логин и пароль, ТО Система ДОЛЖНА сгенерировать JWT_Токен со временем жизни 15 минут
2. КОГДА пользователь отправляет корректные логин и пароль, ТО Система ДОЛЖНА сгенерировать Refresh_Токен со временем жизни 7 дней
3. КОГДА пользователь отправляет некорректные логин или пароль, ТО Система ДОЛЖНА отклонить аутентификацию и вернуть сообщение об ошибке
4. КОГДА пользователь с неподтвержденным телефоном пытается войти, ТО Система ДОЛЖНА отклонить аутентификацию и вернуть сообщение о необходимости подтверждения телефона
5. ГДЕ пользователь выбирает опцию "Запомнить меня", ТО Система ДОЛЖНА увеличить время жизни Refresh_Токена до 30 дней
6. КОГДА пользователь с неактивной учетной записью пытается войти, ТО Система ДОЛЖНА отклонить аутентификацию и вернуть сообщение об ошибке

### Требование 4: Аутентификация через Google OAuth

**Пользовательская история:** Как новый или существующий пользователь, я хочу войти в систему через свой Google аккаунт, чтобы упростить процесс регистрации и входа.

#### Критерии приемки

1. КОГДА пользователь инициирует вход через Google, ТО Система ДОЛЖНА перенаправить пользователя на страницу авторизации Google_OAuth
2. КОГДА пользователь успешно авторизуется в Google, ТО Система ДОЛЖНА получить данные пользователя (email, имя) от Google_OAuth
3. КОГДА пользователь с полученным email не существует в системе, ТО Система ДОЛЖНА создать новую учетную запись с данными из Google_OAuth
4. КОГДА пользователь с полученным email уже существует в системе, ТО Система ДОЛЖНА связать Google аккаунт с существующей учетной записью
5. КОГДА аутентификация через Google_OAuth успешна, ТО Система ДОЛЖНА сгенерировать JWT_Токен и Refresh_Токен
6. КОГДА пользователь впервые входит через Google_OAuth, ТО Система ДОЛЖНА запросить подтверждение телефона через SMS

### Требование 5: Обновление JWT токена

**Пользовательская история:** Как авторизованный пользователь, я хочу автоматически обновлять свою сессию, чтобы не вводить учетные данные повторно при каждом истечении токена.

#### Критерии приемки

1. КОГДА пользователь отправляет валидный Refresh_Токен, ТО Система ДОЛЖНА сгенерировать новый JWT_Токен со временем жизни 15 минут
2. КОГДА пользователь отправляет валидный Refresh_Токен, ТО Система ДОЛЖНА сгенерировать новый Refresh_Токен со временем жизни 7 дней
3. КОГДА пользователь отправляет истекший Refresh_Токен, ТО Система ДОЛЖНА отклонить запрос и вернуть сообщение об ошибке
4. КОГДА пользователь отправляет отозванный Refresh_Токен, ТО Система ДОЛЖНА отклонить запрос и вернуть сообщение об ошибке
5. КОГДА новый Refresh_Токен сгенерирован, ТО Система ДОЛЖНА пометить предыдущий Refresh_Токен как отозванный

### Требование 6: Выход из системы

**Пользовательская история:** Как авторизованный пользователь, я хочу выйти из системы, чтобы завершить свою сессию и обеспечить безопасность аккаунта.

#### Критерии приемки

1. КОГДА пользователь инициирует выход из системы, ТО Система ДОЛЖНА отозвать все активные Refresh_Токены пользователя
2. КОГДА пользователь инициирует выход из системы, ТО Система ДОЛЖНА вернуть подтверждение успешного выхода
3. КОГДА пользователь пытается использовать отозванный Refresh_Токен, ТО Система ДОЛЖНА отклонить запрос

### Требование 7: Генерация надежного пароля

**Пользовательская история:** Как пользователь, я хочу получить предложение надежного пароля, чтобы обеспечить безопасность своего аккаунта.

#### Критерии приемки

1. КОГДА пользователь запрашивает генерацию пароля, ТО Система ДОЛЖНА сгенерировать пароль длиной 16 символов
2. КОГДА Система генерирует пароль, ТО пароль ДОЛЖЕН содержать заглавные буквы, строчные буквы, цифры и специальные символы
3. КОГДА Система генерирует пароль, ТО пароль ДОЛЖЕН быть криптографически случайным

### Требование 8: Управление профилем пользователя

**Пользовательская история:** Как авторизованный пользователь, я хочу просматривать и обновлять свой профиль, чтобы поддерживать актуальность своих данных.

#### Критерии приемки

1. КОГДА авторизованный пользователь запрашивает свой профиль, ТО Система ДОЛЖНА вернуть данные профиля (логин, email, телефон, ФИО)
2. КОГДА пользователь обновляет данные профиля с валидными данными, ТО Система ДОЛЖНА сохранить изменения
3. КОГДА пользователь пытается изменить email на уже существующий, ТО Система ДОЛЖНА отклонить изменение и вернуть сообщение об ошибке
4. КОГДА пользователь изменяет email, ТО Система ДОЛЖНА пометить новый email как неподтвержденный
5. КОГДА пользователь изменяет телефон, ТО Система ДОЛЖНА запросить подтверждение нового телефона через SMS

### Требование 9: Смена пароля

**Пользовательская история:** Как авторизованный пользователь, я хочу изменить свой пароль, чтобы обеспечить безопасность аккаунта.

#### Критерии приемки

1. КОГДА пользователь отправляет текущий пароль и новый пароль, ТО Система ДОЛЖНА проверить корректность текущего пароля
2. КОГДА текущий пароль корректен и новый пароль валиден, ТО Система ДОЛЖНА обновить пароль пользователя
3. КОГДА текущий пароль некорректен, ТО Система ДОЛЖНА отклонить изменение и вернуть сообщение об ошибке
4. КОГДА новый пароль не соответствует требованиям валидации, ТО Система ДОЛЖНА отклонить изменение и вернуть сообщение об ошибке
5. КОГДА пароль успешно изменен, ТО Система ДОЛЖНА отозвать все активные Refresh_Токены пользователя

### Требование 10: Удаление аккаунта

**Пользовательская история:** Как авторизованный пользователь, я хочу удалить свой аккаунт, чтобы прекратить использование системы.

#### Критерии приемки

1. КОГДА пользователь запрашивает удаление аккаунта, ТО Система ДОЛЖНА пометить учетную запись как удаленную (soft delete)
2. КОГДА учетная запись помечена как удаленная, ТО Система ДОЛЖНА отозвать все активные Refresh_Токены пользователя
3. КОГДА учетная запись помечена как удаленная, ТО Система ДОЛЖНА запретить вход с этой учетной записью
4. КОГДА учетная запись помечена как удаленная, ТО Система ДОЛЖНА сохранить данные пользователя в базе данных для аудита

### Требование 11: Интеграция с системой подписок

**Пользовательская история:** Как авторизованный пользователь, я хочу просматривать информацию о своих подписках, чтобы контролировать доступ к платным функциям.

#### Критерии приемки

1. КОГДА авторизованный пользователь запрашивает информацию о подписках, ТО Система ДОЛЖНА получить данные о подписках из nopCommerce
2. КОГДА авторизованный пользователь запрашивает информацию о подписках, ТО Система ДОЛЖНА вернуть список активных и истекших подписок
3. КОГДА пользователь запрашивает доступные планы подписок, ТО Система ДОЛЖНА получить список планов из nopCommerce
4. КОГДА пользователь оформляет подписку, ТО Система ДОЛЖНА создать запись о подписке через nopCommerce
5. КОГДА пользователь отменяет подписку, ТО Система ДОЛЖНА обновить статус подписки через nopCommerce
6. КОГДА пользователь пытается войти в Личный_Кабинет, ТО Система ДОЛЖНА проверить наличие активной подписки

### Требование 12: Хранение паролей

**Пользовательская история:** Как администратор системы, я хочу обеспечить безопасное хранение паролей пользователей, чтобы защитить данные от несанкционированного доступа.

#### Критерии приемки

1. КОГДА пользователь создает или изменяет пароль, ТО Система ДОЛЖНА хешировать пароль с использованием алгоритма PBKDF2
2. Система ДОЛЖНА хранить только хеш пароля, а не пароль в открытом виде
3. КОГДА пользователь аутентифицируется, ТО Система ДОЛЖНА сравнивать хеш введенного пароля с сохраненным хешем

### Требование 13: Защита от брутфорса

**Пользовательская история:** Как администратор системы, я хочу защитить систему от атак перебора паролей, чтобы обеспечить безопасность пользовательских аккаунтов.

#### Критерии приемки

1. КОГДА пользователь совершает более 5 неудачных попыток входа в течение 15 минут, ТО Система ДОЛЖНА временно заблокировать возможность входа для этого пользователя на 15 минут
2. КОГДА IP адрес совершает более 20 неудачных попыток входа в течение часа, ТО Система ДОЛЖНА временно заблокировать запросы с этого IP адреса на 1 час
3. КОГДА период блокировки истекает, ТО Система ДОЛЖНА автоматически разблокировать пользователя или IP адрес

### Требование 14: Логирование операций аутентификации

**Пользовательская история:** Как администратор системы, я хочу отслеживать все операции аутентификации, чтобы обеспечить аудит безопасности.

#### Критерии приемки

1. КОГДА пользователь успешно входит в систему, ТО Система ДОЛЖНА записать событие входа с меткой времени, IP адресом и user agent
2. КОГДА пользователь совершает неудачную попытку входа, ТО Система ДОЛЖНА записать событие с меткой времени, IP адресом и причиной отказа
3. КОГДА пользователь выходит из системы, ТО Система ДОЛЖНА записать событие выхода с меткой времени
4. КОГДА пользователь изменяет пароль, ТО Система ДОЛЖНА записать событие изменения пароля с меткой времени
5. КОГДА происходит подозрительная активность (множественные неудачные попытки входа), ТО Система ДОЛЖНА записать событие с повышенным уровнем важности

### Требование 15: Обработка ошибок и возврат сообщений

**Пользовательская история:** Как пользователь, я хочу получать понятные сообщения об ошибках, чтобы понимать, что пошло не так и как это исправить.

#### Критерии приемки

1. КОГДА происходит ошибка валидации, ТО Система ДОЛЖНА вернуть HTTP статус 400 и описание ошибки на русском языке
2. КОГДА происходит ошибка аутентификации, ТО Система ДОЛЖНА вернуть HTTP статус 401 и общее сообщение об ошибке без раскрытия деталей
3. КОГДА происходит ошибка авторизации, ТО Система ДОЛЖНА вернуть HTTP статус 403 и сообщение о недостаточных правах
4. КОГДА запрашиваемый ресурс не найден, ТО Система ДОЛЖНА вернуть HTTP статус 404 и сообщение о том, что ресурс не найден
5. КОГДА происходит внутренняя ошибка сервера, ТО Система ДОЛЖНА вернуть HTTP статус 500 и общее сообщение об ошибке без раскрытия технических деталей
6. КОГДА происходит любая ошибка, ТО Система ДОЛЖНА записать детали ошибки в лог для последующего анализа

### Требование 16: HTTPS и безопасность транспорта

**Пользовательская история:** Как пользователь, я хочу, чтобы мои данные передавались безопасно, чтобы защитить их от перехвата.

#### Критерии приемки

1. Система ДОЛЖНА принимать запросы только через HTTPS протокол
2. КОГДА клиент отправляет запрос по HTTP, ТО Система ДОЛЖНА перенаправить запрос на HTTPS
3. Система ДОЛЖНА использовать TLS версии 1.2 или выше

### Требование 17: CORS конфигурация

**Пользовательская история:** Как разработчик фронтенда, я хочу, чтобы API поддерживал CORS, чтобы фронтенд приложение могло взаимодействовать с API.

#### Критерии приемки

1. Система ДОЛЖНА разрешать CORS запросы только с доверенных доменов фронтенд приложения
2. КОГДА приходит preflight запрос (OPTIONS), ТО Система ДОЛЖНА вернуть корректные CORS заголовки
3. Система ДОЛЖНА разрешать следующие HTTP методы: GET, POST, PUT, DELETE
4. Система ДОЛЖНА разрешать заголовок Authorization для передачи JWT_Токена

### Требование 18: Health checks и мониторинг

**Пользовательская история:** Как DevOps инженер, я хочу мониторить состояние микросервиса, чтобы обеспечить его доступность.

#### Критерии приемки

1. Система ДОЛЖНА предоставлять endpoint /health для проверки состояния
2. КОГДА все зависимости (база данных, RabbitMQ) доступны, ТО endpoint /health ДОЛЖЕН возвращать HTTP статус 200
3. КОГДА хотя бы одна зависимость недоступна, ТО endpoint /health ДОЛЖЕН возвращать HTTP статус 503
4. Система ДОЛЖНА предоставлять endpoint /health/ready для проверки готовности к обработке запросов
5. Система ДОЛЖНА предоставлять endpoint /health/live для проверки работоспособности процесса

# Документ требований: Course Management API

## Введение

Course Management API - это RESTful веб-сервис на базе ASP.NET Core для управления образовательными курсами и уроками. Система построена на принципах Clean Architecture с использованием Entity Framework Core, PostgreSQL, RabbitMQ и предназначена для развертывания в Kubernetes.

## Глоссарий

- **Система**: Course Management API веб-сервис
- **Курс**: Образовательный курс, содержащий набор уроков
- **Урок**: Отдельное занятие в рамках курса
- **Мягкое_Удаление**: Логическое удаление записи через установку флага Deleted без физического удаления из базы данных
- **Репозиторий**: Компонент для доступа к данным
- **Сервис**: Компонент бизнес-логики
- **DTO**: Объект передачи данных между слоями
- **Миграция**: Скрипт изменения схемы базы данных

## Требования

### Требование 1: Управление курсами

**Пользовательская история:** Как пользователь API, я хочу управлять курсами (создавать, читать, обновлять, удалять), чтобы поддерживать актуальный каталог образовательных программ.

#### Критерии приемки

1. КОГДА пользователь отправляет GET запрос на /Course/{id} с существующим ID, ТО Система ДОЛЖНА вернуть данные курса с HTTP статусом 200
2. КОГДА пользователь отправляет GET запрос на /Course/{id} с несуществующим ID, ТО Система ДОЛЖНА вернуть HTTP статус 404
3. КОГДА пользователь отправляет POST запрос на /Course с валидными данными курса, ТО Система ДОЛЖНА создать новый курс и вернуть его данные с HTTP статусом 201
4. КОГДА пользователь отправляет POST запрос на /Course с невалидными данными, ТО Система ДОЛЖНА вернуть HTTP статус 400 с описанием ошибок валидации
5. КОГДА пользователь отправляет PUT запрос на /Course/{id} с валидными данными, ТО Система ДОЛЖНА обновить существующий курс и вернуть обновленные данные
6. КОГДА пользователь отправляет DELETE запрос на /Course/{id}, ТО Система ДОЛЖНА выполнить Мягкое_Удаление курса через установку флага Deleted в true
7. КОГДА пользователь отправляет POST запрос на /Course/list с параметрами фильтрации, ТО Система ДОЛЖНА вернуть список курсов, соответствующих критериям фильтрации

### Требование 2: Управление уроками

**Пользовательская история:** Как пользователь API, я хочу управлять уроками в рамках курсов, чтобы структурировать образовательный контент.

#### Критерии приемки

1. КОГДА пользователь отправляет GET запрос на /Lesson/{id} с существующим ID, ТО Система ДОЛЖНА вернуть данные урока с HTTP статусом 200
2. КОГДА пользователь отправляет GET запрос на /Lesson/{id} с несуществующим ID, ТО Система ДОЛЖНА вернуть HTTP статус 404
3. КОГДА пользователь отправляет POST запрос на /Lesson с валидными данными урока, ТО Система ДОЛЖНА создать новый урок и вернуть его данные с HTTP статусом 201
4. КОГДА пользователь отправляет POST запрос на /Lesson с невалидным CourseId, ТО Система ДОЛЖНА вернуть HTTP статус 400
5. КОГДА пользователь отправляет PUT запрос на /Lesson/{id} с валидными данными, ТО Система ДОЛЖНА обновить существующий урок и вернуть обновленные данные
6. КОГДА пользователь отправляет DELETE запрос на /Lesson/{id}, ТО Система ДОЛЖНА выполнить Мягкое_Удаление урока через установку флага Deleted в true
7. КОГДА пользователь отправляет GET запрос на /Lesson/list/{page}/{itemsPerPage}, ТО Система ДОЛЖНА вернуть постраничный список уроков с указанной страницей и количеством элементов

### Требование 3: Валидация данных

**Пользовательская история:** Как разработчик системы, я хочу обеспечить валидацию входных данных, чтобы поддерживать целостность данных в базе.

#### Критерии приемки

1. КОГДА создается или обновляется Курс с полем Name длиной более 100 символов, ТО Система ДОЛЖНА отклонить операцию с HTTP статусом 400
2. КОГДА создается или обновляется Курс с пустым полем Name, ТО Система ДОЛЖНА отклонить операцию с HTTP статусом 400
3. КОГДА создается или обновляется Урок с полем Subject длиной более 100 символов, ТО Система ДОЛЖНА отклонить операцию с HTTP статусом 400
4. КОГДА создается или обновляется Урок с пустым полем Subject, ТО Система ДОЛЖНА отклонить операцию с HTTP статусом 400
5. КОГДА создается Урок с несуществующим CourseId, ТО Система ДОЛЖНА отклонить операцию с HTTP статусом 400
6. КОГДА создается или обновляется Курс с отрицательным значением Price, ТО Система ДОЛЖНА отклонить операцию с HTTP статусом 400

### Требование 4: Мягкое удаление

**Пользовательская история:** Как администратор системы, я хочу использовать механизм мягкого удаления, чтобы сохранять историю данных и иметь возможность восстановления.

#### Критерии приемки

1. КОГДА выполняется удаление Курса, ТО Система ДОЛЖНА установить флаг Deleted в true вместо физического удаления записи
2. КОГДА выполняется удаление Урока, ТО Система ДОЛЖНА установить флаг Deleted в true вместо физического удаления записи
3. КОГДА выполняется запрос списка курсов, ТО Система ДОЛЖНА исключить курсы с флагом Deleted равным true
4. КОГДА выполняется запрос списка уроков, ТО Система ДОЛЖНА исключить уроки с флагом Deleted равным true
5. КОГДА выполняется запрос курса по ID с флагом Deleted равным true, ТО Система ДОЛЖНА вернуть HTTP статус 404

### Требование 5: Интеграция с базой данных

**Пользовательская история:** Как DevOps инженер, я хочу автоматизировать управление схемой базы данных, чтобы упростить развертывание и обновление системы.

#### Критерии приемки

1. КОГДА Система запускается с настройкой Database:AutoMigrate равной true, ТО Система ДОЛЖНА автоматически применить все pending миграции к базе данных
2. КОГДА Система не может подключиться к базе данных, ТО Система ДОЛЖНА повторить попытку подключения от 3 до 5 раз с задержкой
3. КОГДА все попытки подключения к базе данных исчерпаны, ТО Система ДОЛЖНА завершить запуск с ошибкой
4. Система ДОЛЖНА создать индекс на поле Name в таблице Courses для оптимизации поиска
5. Система ДОЛЖНА создать индекс на поле Subject в таблице Lessons для оптимизации поиска
6. КОГДА выполняется запрос к /diagnostics/migrations, ТО Система ДОЛЖНА вернуть список примененных и pending миграций

### Требование 6: Обмен сообщениями

**Пользовательская история:** Как архитектор системы, я хочу публиковать события о важных операциях, чтобы другие сервисы могли реагировать на изменения.

#### Критерии приемки

1. КОГДА создается новый Курс, ТО Система ДОЛЖНА опубликовать событие в RabbitMQ с данными созданного курса
2. КОГДА публикация события в RabbitMQ не удается, ТО Система ДОЛЖНА залогировать ошибку и продолжить выполнение операции
3. КОГДА обновляется Курс, ТО Система ДОЛЖНА опубликовать событие обновления в RabbitMQ
4. КОГДА удаляется Курс, ТО Система ДОЛЖНА опубликовать событие удаления в RabbitMQ

### Требование 7: Мониторинг и диагностика

**Пользовательская история:** Как DevOps инженер, я хочу иметь endpoints для проверки здоровья системы, чтобы мониторить её состояние в production.

#### Критерии приемки

1. КОГДА выполняется запрос к /health, ТО Система ДОЛЖНА вернуть HTTP статус 200 если система работает корректно
2. КОГДА выполняется запрос к /db_ef_healthcheck, ТО Система ДОЛЖНА проверить подключение к базе данных и вернуть соответствующий статус
3. КОГДА база данных недоступна, ТО endpoint /db_ef_healthcheck ДОЛЖЕН вернуть HTTP статус 503
4. КОГДА выполняется запрос к /diagnostics/config, ТО Система ДОЛЖНА вернуть текущую конфигурацию (без чувствительных данных)
5. Система ДОЛЖНА поддерживать readiness probe для Kubernetes на endpoint /health
6. Система ДОЛЖНА поддерживать liveness probe для Kubernetes на endpoint /health

### Требование 8: Конфигурация и окружение

**Пользовательская история:** Как DevOps инженер, я хочу настраивать систему через переменные окружения, чтобы использовать одинаковый образ в разных средах.

#### Критерии приемки

1. КОГДА установлена переменная окружения DB_HOST, ТО Система ДОЛЖНА использовать её значение для подключения к базе данных
2. КОГДА установлена переменная окружения DB_PORT, ТО Система ДОЛЖНА использовать её значение для подключения к базе данных
3. КОГДА установлена переменная окружения DB_NAME, ТО Система ДОЛЖНА использовать её значение как имя базы данных
4. КОГДА установлена переменная окружения DB_USER, ТО Система ДОЛЖНА использовать её значение для аутентификации
5. КОГДА установлена переменная окружения DB_PASSWORD, ТО Система ДОЛЖНА использовать её значение для аутентификации
6. КОГДА установлена переменная окружения DB_SSL_MODE, ТО Система ДОЛЖНА использовать её значение для настройки SSL подключения
7. ГДЕ окружение Development, Система ДОЛЖНА логировать SQL запросы для отладки

### Требование 9: API документация

**Пользовательская история:** Как разработчик клиентского приложения, я хочу иметь интерактивную документацию API, чтобы быстро понять доступные endpoints и их параметры.

#### Критерии приемки

1. Система ДОЛЖНА предоставлять Swagger UI для интерактивной документации API
2. КОГДА пользователь открывает Swagger UI, ТО Система ДОЛЖНА отобразить все доступные endpoints с их параметрами
3. КОГДА пользователь выполняет запрос через Swagger UI, ТО Система ДОЛЖНА обработать его как обычный API запрос
4. Swagger документация ДОЛЖНА включать описания всех DTO моделей
5. Swagger документация ДОЛЖНА включать примеры запросов и ответов

### Требование 10: Архитектурная организация

**Пользовательская история:** Как разработчик системы, я хочу следовать принципам Clean Architecture, чтобы обеспечить поддерживаемость и тестируемость кода.

#### Критерии приемки

1. КОГДА изменяется доменная логика, ТО слой Domain НЕ ДОЛЖЕН зависеть от слоев Infrastructure, Services или WebApi
2. КОГДА изменяется способ хранения данных, ТО слой Infrastructure ДОЛЖЕН быть изолирован от бизнес-логики
3. Система ДОЛЖНА использовать паттерн Repository для доступа к данным
4. Система ДОЛЖНА использовать слой Services для бизнес-логики
5. Система ДОЛЖНА использовать DTO для передачи данных между слоями
6. Система ДОЛЖНА использовать AutoMapper для маппинга между доменными моделями и DTO
7. Система ДОЛЖНА использовать Dependency Injection для управления зависимостями

### Требование 11: CORS и безопасность

**Пользовательская история:** Как разработчик фронтенд приложения, я хочу иметь возможность обращаться к API из браузера, чтобы создавать веб-интерфейсы.

#### Критерии приемки

1. Система ДОЛЖНА поддерживать CORS запросы от всех источников
2. КОГДА выполняется preflight OPTIONS запрос, ТО Система ДОЛЖНА вернуть корректные CORS заголовки
3. КОГДА выполняется запрос с заголовком Origin, ТО Система ДОЛЖНА включить заголовок Access-Control-Allow-Origin в ответ

### Требование 12: Развертывание в Kubernetes

**Пользовательская история:** Как DevOps инженер, я хочу развернуть систему в Kubernetes с высокой доступностью, чтобы обеспечить надежность сервиса.

#### Критерии приемки

1. Система ДОЛЖНА поддерживать развертывание в Kubernetes с минимум 2 репликами
2. КОГДА pod запускается, ТО Kubernetes ДОЛЖЕН использовать readiness probe для определения готовности
3. КОГДА pod работает, ТО Kubernetes ДОЛЖЕН использовать liveness probe для проверки здоровья
4. КОГДА readiness probe возвращает ошибку, ТО Kubernetes НЕ ДОЛЖЕН направлять трафик на этот pod
5. КОГДА liveness probe возвращает ошибку несколько раз подряд, ТО Kubernetes ДОЛЖЕН перезапустить pod
